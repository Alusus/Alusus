Alusus Building and Usage Instructions

External Dependencies

Following is the list of dependencies you'll need to build the project from the source code.

LLVM 3.3

Boost 1.55.0

CATCH

The release of CATCH used by Alusus' source code is old, therefore you need to download CATCH from the link above and not from their official page which includes a newer release that might not be compatible with Alusus' source code. In addition to the libraries mentioned above, you'll need GCC (4.8) and CMake to be able to build the binaries from the source code. LLVM also has its own dependencies like valgrind for example.

Building Steps

  • Install the packages for build tools like GCC and CMake using your distribution's package manager. Depending on the distribution you are using, some of these packages might already be installed for you.
    In Ubuntu use the following command: 
    $ sudo apt-get install gcc cmake
    And for Fedora use the following: 
    $ sudo yum install gcc gcc-c++ cmake valgrind valgrind-devel
    Notice the need to install valgrind in Fedora in order to be able to build LLVM. We didn't need to install valgrind in Ubuntu. You might also need to install other packages if you are using distros other than the above. Visit LLVM's website for more details on how to build it.
    Note: You'll need GCC version 4.8 or newer. You won't be able to build Alusus code with older versions.
  • Download the source code for the versions mentioned above of LLVM, Boost, and CATCH, and extract their files.
  • Build and install LLVM:
    • Create a build folder with any name. For example: llvm.build.
    • CD into the build folder you created and execute the following command: 
      $ cmake <path-to-LLVM-source>
      You might also need to manually specify your architecture by setting a value for LLVM_TARGET_ARCH. For example, to specify the architecture under a 32 bit environment use the following command instead of the one above: 
      $ cmake <path-to-LLVM-source> -DLLVM_TARGET_ARCH=i386
      and to specify it for a 64 bit environment use the following: 
      $ cmake <path-to-LLVM-source> -DLLVM_TARGET_ARCH=x86_64
    • Complete the build and install: 
      $ make
$ sudo make install
  • Build Alusus:
    • Created a build folder and name it anything. For example, Builds.
    • CD into that folder and complete the build: 
      $ cd <path-to-Builds>
$ cmake <path-to-Alusus-root>/Sources -DBOOST_PATH=<path-to-Boost-source> -DCATCH_PATH=<path-to-CATCH> -DCMAKE_BUILD_TYPE=debug -DCMAKE_INSTALL_PREFIX=<path-to-Alusus-root> -DLLVM_PATH=<path-to-LLVM-build>
$ make
$ make install
      The last statement (make install) will install the build results into folders Bin, Lib, and Include under Alusus' root folder, which is mentioned above in CMAKE_INSTALL_PREFIX cmake parameter, and not to the operating system's paths. You can choose a different folder as an installation target like /opt/Alusus by changing the value for CMAKE_INSTALL_PREFIX. The above cmake command prepares for a debug build. To change to a release build replace the value of CMAKE_BUILD_TYPE with release instead of debug.

    Executing Automated Tests

    Executing the automated tests is simple. All you need to do is to run the following command inside the Alusus' build folder (Builds in the example above) after the build is complete: 
    $ make test
    This command will run three tests. You can run each of these tests individually by changing into its directory under Builds/Tests and manually running its executable. The test CoreTests can be controlled through command line arguments which you can get details on by passing the --help argument.

    Using Alusus

    After the build is complete you'll have the executable under Bin and the libraries will go under Lib. In order to run the core you'll need to add the Lib folder to LD_LIBRARY_PATH. After that you'll be able to run the examples: 
    $ export LD_LIBRARY_PATH=<path-to-Alusus>/Lib:$LD_LIBRARY_PATH
$ export PATH=<path-to-Alusus>/Bin:PATH
$ cd <path-to-Alusus>/Examples
$ alusus_d helloworld.alusus
    The Core's executalbe's name when building in debug mode will be alusus_d. If the build is in release mode the name will be alusus, i.e. without the _d extension.

    If the Core is built in debug mode logging can be enabled using the --log command line argument, which accepts a number representing the details level of the log as in the following example: 

    $ alusus_d helloworld.alusus --log 16
    The details level is determined through binary flags representing each different details level with the least significant bit representing the lowest level. A value of 1 for each level means details for that level will be logged to the screen. The following table explains the different levels:
    Lexer, low level = 1
    Lexer, mid level = 2
    Lexer, high level = 4
    Parser, low level = 8
    Parser, mid level = 16
    Parser, high level = 32
    The flags can be combined to display more than one level at the same time. For example, a value of 7 means logging all levels for the lexer.

    It's also possible to enable logging in the same way for CoreTests.
